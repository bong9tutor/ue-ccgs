기술 설정 & 사전 제작 — 구현 준비

설계 문서를 실제 구현 계획으로 변환합니다. 아키텍처 결정, 에픽/스토리 분해, 스프린트 계획까지

1

/create-architecture — 마스터 아키텍처 문서 만들기

"아키텍처"가 뭔가요?

건물을 지을 때 "기둥이 어디 가고, 배관은 어디로 지나가는지"를 정하는 구조 설계도가 있듯이, 게임도 "전투 시스템이 체력 시스템과 어떻게 연결되고, 데이터는 어디에 저장되는지"를 미리 정합니다. 이것이 아키텍처입니다.

Claude에게 아키텍처 생성 요청하기

Claude가 Part 3에서 작성한 모든 GDD를 읽고, 전체 게임의 기술 구조를 설계합니다.

아키텍처 생성
나 > /create-architecture Claude (Tech Lead): "모든 GDD를 분석해서 마스터 아키텍처를 작성합니다." "분석할 GDD: combat.md, health.md, inventory.md, save-load.md..." ━━ UE5.6 기술 스택 선택 ━━ Claude: "이 게임에 최적인 UE5.6 시스템들을 추천합니다:" ┌─────────────────┬───────────────────────────────┐ │ 기능 │ UE5.6 시스템 │ ├─────────────────┼───────────────────────────────┤ │ 전투/스킬 │ Gameplay Ability System (GAS) │ │ 입력 처리 │ Enhanced Input System │ │ UI/메뉴 │ CommonUI + UMG │ │ 조명/렌더링 │ Lumen (글로벌 일루미네이션) │ │ 메시/지오메트리 │ Nanite (자동 LOD) │ │ 이펙트 │ Niagara (파티클 시스템) │ │ 사운드 │ MetaSounds │ └─────────────────┴───────────────────────────────┘ "동의하시나요? 수정할 부분이 있으면 말씀해주세요." 나 > 좋아요, 그대로 진행해주세요 ━━ 모듈 구조 ━━ Claude: "게임을 다음 모듈로 나눕니다:" Core — 게임 기본 설정, 데이터 타입 Combat — GAS 기반 전투, 콤보, 데미지 Character — 플레이어/NPC 캐릭터 로직 UI — HUD, 메뉴, 인벤토리 화면 World — 레벨, 스폰, 환경 SaveSystem — 저장/불러오기 ✓ architecture/master-architecture.md 생성 완료 ✓ architecture/module-dependency-graph.md 생성 완료
💡
이 용어들을 몰라도 괜찮습니다!

Claude가 각 기술이 왜 필요한지 쉬운 말로 설명해줍니다. 예: "GAS는 레고 블록처럼 스킬을 조립하는 시스템"이라고 설명합니다.

2

/architecture-decision — 기술 결정 기록하기

"ADR"이 뭔가요?

Architecture Decision Record의 약자로, "왜 이 기술을 골랐는가"를 기록하는 문서입니다. 나중에 "왜 이렇게 만들었지?" 할 때 되돌아볼 수 있습니다.

기술 결정 예시: GAS 선택

ADR 작성
나 > /architecture-decision "전투 시스템에 GAS 사용" Claude: "ADR을 작성합니다. 몇 가지 질문이 있어요." ━━ 옵션 비교 ━━ Claude: "전투 시스템 구현에 3가지 방법이 있습니다:" A) GAS (Gameplay Ability System) 장점: 버프/디버프 자동 관리, 멀티플레이 대응, 에픽 공식 지원 단점: 초기 설정이 복잡함 B) 커스텀 전투 시스템 (직접 구현) 장점: 단순하게 시작 가능 단점: 기능 추가할수록 복잡해짐, 버그 위험 C) 플러그인 사용 (마켓플레이스) 장점: 빠른 시작 단점: 커스터마이징 제한, 유지보수 어려움 "어떤 것을 선택하시겠어요?" 나 > A. GAS로 해주세요 Claude: "좋은 선택입니다! ADR을 기록합니다:" 결정: GAS 사용 이유: 장기적 확장성과 멀티플레이 대응 결과: 초기 설정에 1-2일 추가 소요 예상 ✓ architecture/adr/ADR-001-combat-gas.md 저장 완료
📝

중요한 기술 결정이 있을 때마다 /architecture-decision을 실행합니다. 예: 입력 시스템 선택, 네트워크 방식, 저장 포맷 등. Claude가 매번 장단점을 비교해줍니다.

/create-epics — GDD를 에픽으로 분해하기

"에픽"이 뭔가요?

에픽(Epic)은 "전투 시스템 전체 구현"처럼 큰 덩어리의 작업입니다. 너무 커서 한 번에 못 끝내기 때문에, 나중에 더 작은 조각(스토리)으로 나눕니다. 책으로 치면 "챕터" 같은 것입니다.

에픽 생성하기

에픽 분해
나 > /create-epics Claude: "모든 GDD를 분석해서 에픽을 생성합니다..." ━━ 에픽 목록 (우선순위별) ━━ EP-001 │ 플레이어 이동 & 카메라 │ P0 (MVP) │ ~5 스토리 EP-002 │ 기본 전투 (GAS) │ P0 (MVP) │ ~8 스토리 EP-003 │ 체력 & 데미지 시스템 │ P0 (MVP) │ ~4 스토리 EP-004 │ 인벤토리 기본 │ P0 (MVP) │ ~5 스토리 EP-005 │ 적 AI & 스폰 │ P1 │ ~6 스토리 EP-006 │ UI & HUD │ P1 │ ~7 스토리 EP-007 │ 보스 전투 │ P1 │ ~4 스토리 EP-008 │ 저장/불러오기 │ P1 │ ~3 스토리 EP-009 │ 사운드 & 이펙트 │ P2 │ ~5 스토리 EP-010 │ 레벨 디자인 │ P2 │ ~6 스토리 ─────────────────────────────────────────────────── 합계: 10 에픽, ~53 스토리 예상 "이 구성이 맞나요? 추가/수정할 에픽이 있으면 말씀해주세요." 나 > 좋아요! ✓ backlog/epics.md 생성 완료
🤖
P0/P1/P2는 우선순위입니다:

P0 (MVP) = 게임이 돌아가려면 반드시 필요, P1 = 있으면 훨씬 좋음, P2 = 나중에 추가해도 됨

/create-stories — 에픽을 스토리로 쪼개기

"스토리"가 뭔가요?

스토리(Story)는 "경공격 3타 콤보 구현"처럼 하루 이틀 안에 끝낼 수 있는 작은 작업입니다. 에픽이 "챕터"라면, 스토리는 그 안의 "한 페이지"입니다. Claude가 실제로 코딩하는 단위가 바로 이 스토리입니다.

에픽을 스토리로 분해하기

스토리 생성
# EP-002 (기본 전투)를 스토리로 쪼개기 나 > /create-stories EP-002 Claude: "EP-002 (기본 전투 시스템)의 스토리를 생성합니다:" ━━ EP-002의 스토리 목록 ━━ ST-001 │ GAS 플러그인 활성화 & 기본 클래스 │ 낮음 │ 1일 ST-002 │ 경공격 3타 콤보 구현 │ 중간 │ 2일 ST-003 │ 중공격 & 차지 어택 구현 │ 중간 │ 2일 ST-004 │ 회피 (대시/구르기) 구현 │ 낮음 │ 1일 ST-005 │ 패리 시스템 (타이밍 기반) │ 높음 │ 3일 ST-006 │ 잡기 공격 구현 │ 중간 │ 2일 ST-007 │ 히트 리액션 & 넉백 │ 중간 │ 2일 ST-008 │ 콤보 카운터 UI 연동 │ 낮음 │ 1일 "각 스토리에 수용 기준(Acceptance Criteria)도 작성되어 있어요." /sprint-plan new — 첫 번째 스프린트 계획하기

"스프린트"가 뭔가요?

스프린트(Sprint)는 1~2주 동안 집중해서 작업하는 기간입니다. 마라톤처럼 한 번에 달리는 게 아니라, 짧은 구간을 전력 질주하듯 작은 목표를 정해서 완성합니다. 스프린트가 끝나면 결과를 확인하고, 다음 스프린트를 계획합니다.

첫 스프린트 계획 세우기

스프린트 계획
나 > /sprint-plan new Claude: "스프린트 1을 계획합니다." 기간: 2주 목표: "플레이어가 움직이고 기본 공격을 할 수 있다" ━━ 스프린트 1 스토리 선택 ━━ ✓ ST-001 │ GAS 기본 설정 │ 1일 │ EP-002 ✓ ST-002 │ 경공격 3타 콤보 │ 2일 │ EP-002 ✓ EP1-ST-001 │ 플레이어 이동 │ 2일 │ EP-001 ✓ EP1-ST-002 │ 카메라 시스템 │ 1일 │ EP-001 ✓ EP3-ST-001 │ 체력 컴포넌트 │ 1일 │ EP-003 ───────────────────────────────────────── 합계: 5개 스토리, 약 7일 작업량 "의존성 확인: ST-002는 ST-001 완료 후 시작합니다." "이 계획으로 진행할까요?" 나 > 네, 시작합시다! ✓ sprints/sprint-001.md 저장 완료
⚠️
의존성이 중요합니다!

Claude는 자동으로 "이 스토리가 먼저 끝나야 저 스토리를 시작할 수 있다"는 순서를 파악합니다. 예: GAS 설정(ST-001)이 끝나야 콤보(ST-002)를 만들 수 있습니다.
